# 같이하기 이름 변경 + 간격 조정 — 설계

작성일: 2026-07-21
대상: `public/index.html` (같이하기 멀티플레이 모드)

## 배경

같이하기 모드의 닉네임은 최초 입장 때 한 번만 묻고 `localStorage`(`NICK_KEY = "nachmal.multi.nick.v1"`)에
저장된다. 한 번 정하면 바꿀 방법이 없고, 내 칩은 이름 대신 `나`로만 표시되어 내가 남들에게 어떤
이름으로 보이는지 확인할 수조차 없다.

함께 두 건의 간격 문제도 고친다. 둘 다 같은 화면의 시각적 결함이라 한 작업으로 묶는다.

## 목표

1. 같이하기 화면에서 언제든 이름을 바꿀 수 있다.
2. 내 칩에서 내 현재 이름을 확인할 수 있다.
3. 이름 칩 목록과 문제 카드 사이, 교체 요청 확인 카드의 위아래 간격을 정상화한다.

## 비목표

- 이름 변경을 다른 플레이어에게 알리지 않는다(🦉 말풍선은 문제·정답·교체투표 중계 전용이며,
  이름 알림이 그 위를 덮으면 레이스 중 정보를 가린다).
- 중복 이름을 막지 않는다(기존 동작 유지).
- 혼자하기 모드는 건드리지 않는다.

## 1. 데이터 흐름

이름은 `nunus/<mode>/presence/<uid>/name`에만 존재하고 문제·점수·판정 로직과 완전히 분리되어
있다. 따라서 변경은 그 한 필드의 `update`가 전부다.

```
사용자가 이름 확정
  → localStorage 갱신 (다음 입장에 이어짐)
  → MG.name 갱신
  → presence/<uid> 에 { name } update   ← ts는 건드리지 않는다
  → presence on('value') 가 모든 참가자에게 전파
  → 각자 renderPlayers() 가 칩을 다시 그림
```

별도 브로드캐스트 경로를 만들지 않는다. 이미 있는 presence 구독이 전파를 담당한다.

`set`이 아니라 `update`를 쓴다. `set`은 `ts`를 지워 재접속 판정 데이터를 잃는다.

## 2. 진입점과 칩 표시

`renderPlayers()`에서 내 칩만 다르게 그린다.

- 내 칩: `이름(나) ✏️` + 점수. 클릭 핸들러를 생성 시점에 부착한다(칩은 매 렌더마다 새로 만들어진다).
- 남의 칩: 기존 그대로.

이름이 비어 있는 이론적 경우에는 `나`만 표시한다. 이름은 기존 남의 칩과 동일하게 `escapeHtml`을
거친다.

## 3. 편집 UI

기존 `#multi-nick` 블록을 재사용한다. 말풍선 바로 아래, 게임 영역 위에 이미 자리잡고 있어
인라인 편집 위치로 그대로 맞는다. 입력 검증·Enter 키 바인딩·`maxlength=12`를 모두 물려받는다.

`askNick(mode)`로 두 모드를 구분한다.

| | 최초 입장 (`enter`) | 이름 변경 (`edit`) |
|---|---|---|
| 입력 초기값 | 빈 값 | 현재 이름 |
| 확인 버튼 라벨 | 입장 | 바꾸기 |
| 취소 버튼 | 없음 (이름 없이는 입장 불가) | 있음 |
| `#multi-play` | 숨김 | 유지 (문제가 계속 보인다) |
| 말풍선 | "같이 하려면 이름이 필요해!" | "새 이름을 뭐라고 할까?" |

새로 추가되는 마크업은 취소 버튼 하나뿐이다. `enter` 모드에서는 숨긴다.

## 4. 간섭 차단 — 핵심 위험 지점

`renderMulti()`는 presence·meta·scores 세 리스너와 1초 tick에서 반복 호출되고, `play` 분기에서
게임 입력창을 다시 켜고 말풍선을 덮어쓴다. 게이팅이 없으면 편집 중 라운드가 넘어가는 순간
게임 입력창이 되살아나 두 입력창이 경합하고 안내 문구가 사라진다.

이는 `progress.md`에 기록된 오프라인 게이팅 회귀(20e787b)와 정확히 같은 구조다.

**대책**: `MG.nickEditing` 플래그를 두고 `renderMulti()`의 `play` 분기 활성화 조건을
`online && !MG.nickEditing`으로 묶는다.

- 편집 중: 게임 입력창은 잠긴 채 유지되고 편집 안내 문구가 살아남는다.
- 편집 종료(확정·취소 모두): 플래그를 내리고 `renderMulti()`를 한 번 호출해 정상 복귀한다.

`reveal`/`over` 분기는 이미 입력을 잠그므로 게이팅하지 않는다. 편집 중 게임이 `over`로 넘어가면
`#multi-play`가 숨겨지지만 `#multi-nick`은 그 바깥이라 편집창은 그대로 남는다 — 종료 화면에서도
이름 변경은 유효하므로 의도된 동작이다.

## 5. 차단 조건

다음 두 경우에는 칩 탭을 무시한다(편집을 열지 않는다).

- `MG.online === false` — 쓰기가 조용히 버려진다.
- 교체 투표 진행 중(`MG.meta.swap`) — 이미 입력이 잠긴 구간이다.

편집 중에 연결이 끊기면(`onCon`의 오프라인 분기) 편집을 닫고 플래그를 내린다. 오프라인 잠금
로직이 편집 UI를 모른 채 입력만 잠그면 편집창이 떠 있는 채로 남는다.

## 6. 검증 규칙

기존 `submitNick`과 동일한 규칙을 공유한다.

- `norm()` 후 12자로 자른다.
- 빈 값·공백만 → 닫지 않고 입력에 포커스를 유지한다.
- 값이 현재 이름과 같으면 → 쓰기 없이 그냥 닫는다.
- 중복 이름 허용.

## 7. 간격 조정

두 건 모두 CSS 값 수정이다.

| 대상 | 현재 | 변경 | 이유 |
|---|---|---|---|
| `.players` `padding` | `10px 16px 2px` | `10px 16px 12px` | `.wrap`에 gap이 없어 이 padding-bottom이 칩 목록과 문제 카드 사이의 유일한 간격이다. 칩의 `box-shadow` 여유도 함께 확보된다. |
| `.swap-card` `margin` | `4px 0 2px` | `14px 0 12px` | `#multi-play`는 일반 블록이라 margin이 그대로 적용된다. 위쪽 문제 영역, 아래쪽 입력창과 모두 붙어 있다. |

`.players`는 `#multi-players` 한 곳에서만 쓰이므로 다른 화면에 영향이 없다.

## 8. 테스트

**순수 로직** — `test/logic.test.mjs`

- 이름 정규화: 앞뒤 공백 제거, 12자 절단, 공백만 입력은 거부.
- "변경 없음" 판정: 정규화 결과가 현재 이름과 같으면 쓰기를 건너뛴다.

**통합** — `test/multi-race.mjs` (puppeteer 2탭)

시나리오 하나를 추가한다. 목적은 4번 게이팅을 fail-before/pass-after로 증명하는 것이다.

1. 탭A·탭B 입장, 라운드 진행 중.
2. 탭A가 칩을 탭해 이름을 바꾼다.
3. 탭B의 칩 목록에 새 이름이 나타난다.
4. 그동안 탭A의 게임 입력창은 잠겨 있다.
5. 편집을 닫으면 입력창이 되살아나고 실제 제출이 성공한다.

게이팅을 제거한 상태에서 4번이 실패하는 것을 먼저 확인한 뒤 통과시킨다.

**회귀**: 기존 66개 테스트 전량 통과, 혼자하기 모드 무손상 확인.
